### Concurrency model

watax serves with a **reactor pool** (`listen_reactor_pool`): a small, fixed
set of OS worker threads, each running its OWN independent readiness reactor
(epoll/kqueue/WSAPoll) over its own connection table — no mutable state is
shared between workers, so it is data-race-free and passes `--strict`. Every
connection is one fd in a worker's readiness set doing non-blocking I/O —
concurrency without a thread per request, the same model as axum/tokio.
Request headers are parsed lazily (scanned on demand) and all
per-request memory is freed inside the framework (auto-drop + owning response
APIs), so peak RSS stays flat under sustained load — user handlers never call
free/dispose.

### Test types

Beyond plaintext/JSON, the suite includes TechEmpower-style endpoints:
**db** (single row), **queries** (N rows), **updates** (N rows read +
written), and **fortunes** (HTML table with per-row escaping). These use an
**in-memory** `World`/`Fortune` store — there is no real database, so they
measure framework overhead (routing, query-string parsing, JSON
serialization, HTML templating), not DB latency. watax additionally serves a
WebSocket echo at `/ws` (RFC 6455), which isn't in the table above because
`wrk` benchmarks HTTP, not WebSocket. The watax app is built with `-O3`.
